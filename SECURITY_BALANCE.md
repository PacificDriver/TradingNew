# Безопасность балансов

Анализ защиты от несанкционированного изменения баланса и типичных атак.

---

## Что сделано хорошо

1. **Авторизация**
   - Все операции с балансом требуют JWT (`authMiddleware`). `userId` берётся из токена, не из тела запроса.
   - Вывод и открытие сделок дополнительно проверяют блокировки (`requireNotBlockedMiddleware`, `withdrawBlockedAt`).

2. **Единственный источник пополнения**
   - Увеличение баланса возможно только через:
     - колбек HighHelp (payin success),
     - возврат при отказе/ошибке вывода (payout decline/error),
     - начисление выигрыша по сделке (расчёт на сервере),
     - перевод реферального баланса в основной (по кнопке пользователя).
   - Нет публичного API «добавить баланс» без привязки к платежу или сделке.

3. **Идемпотентность колбеков**
   - Повторная обработка одного и того же колбека HighHelp исключена через `ProcessedCallback` (ключ `project_id:payment_id:status:sub_status`).

4. **Вывод**
   - Баланс списывается в одной транзакции с созданием `PaymentTransaction`. При ошибке `createPayout` баланс возвращается в `catch`.

## Уязвимости и минусы

### 1. **Гонка при открытии сделки (критично)**

**Проблема:** Проверка `user.demoBalance >= amount` выполняется по данным, прочитанным до транзакции. В транзакции баланс обновляется как `Number(user.demoBalance) - amount`, т.е. снова используется старое значение. При двух одновременных запросах оба могут пройти проверку и оба спишут с одного и того же «старого» баланса → возможен двойной спенд (открытие сделок на сумму больше фактического баланса).

**Рекомендация:** Атомарно списывать в БД и проверять достаточность баланса в одном запросе.  
**Исправлено:** используется `UPDATE "User" SET "demoBalance" = "demoBalance" - $amount WHERE id = $userId AND "demoBalance" >= $amount`; при 0 обновлённых строк возвращается 400. Создание сделки — в той же транзакции.

### 2. **Вебхук HighHelp без проверки подписи**

**Исправлено:** перед начислением по колбеку выполняется проверка:
1. **project_id** — в теле колбека должен совпадать с `HIGHHELP_PROJECT_ID`. Иначе возвращается 401, баланс не меняется.
2. **Подпись (по документации HighHelp):**
   - **Предпочтительно:** задайте **HIGHHELP_CALLBACK_PUBLIC_KEY_PEM** — публичный ключ из ЛК HighHelp: «API» → «Настройки Callback» → блок «Public Key», скачать. Колбеки HighHelp подписываются RSA (как исходящие запросы: normalize(body) → base64url + timestamp → SHA256 → подпись). Проверяются заголовки **x-access-signature** и **x-access-timestamp**.
   - **Альтернатива:** задайте **HIGHHELP_WEBHOOK_SECRET** — тогда проверяется заголовок **X-Webhook-Secret** (равен секрету) или **X-HighHelp-Signature** / **X-Signature** = HMAC-SHA256(raw body, секрет) в hex.
Если ни ключ, ни секрет не заданы, принимаются колбеки только с верным `project_id`. В проде рекомендуется задать `HIGHHELP_CALLBACK_PUBLIC_KEY_PEM`. Тело вебхука читается в raw-виде. При неверной подписи — 401, баланс не меняется.

### 3. **Гонка при расчёте экспирации (двойное начисление выигрыша)**

**Проблема:** `settleExpiredTrades()` раз в 3 секунды выбирает все активные сделки с истёкшим временем. Нет блокировки по сделке: два параллельных запуска (или повторный запуск до записи в БД) могут обработать одну и ту же сделку дважды и дважды начислить выигрыш.

**Рекомендация:** В одной транзакции обновлять сделку только если она ещё в статусе ACTIVE.  
**Исправлено:** используется `updateMany({ where: { id, status: ACTIVE }, data: { status, closePrice } })`; начисление баланса/реферала выполняется только если `count === 1`, в той же транзакции.

### 4. **Гонка при выводе реферального баланса**

**Проблема:** Сумма к выводу берётся из `user.referralBalance`, после чего выполняется один `update` (обнуление реферального и увеличение основного баланса). При двух одновременных запросах оба могут прочитать один и тот же `referralBalance` и оба выполнить обновление → двойное зачисление на основной баланс.

**Рекомендация:** Делать атомарный перевод в одном запросе.  
**Исправлено:** используется один `UPDATE "User" SET "demoBalance" = "demoBalance" + "referralBalance", "referralBalance" = 0 WHERE id = $id AND "referralBalance" > 0`; при 0 обновлённых строк возвращается 400.

### 5. **JWT и сессии**

- **Слабый JWT_SECRET по умолчанию** (`supersecretjwtkey`) — в проде обязательно задать свой длинный случайный секрет.
- Истечение срока жизни токена и отзыв сессий (logout) реализованы; при компрометации токена злоумышленник может выполнять действия от имени пользователя до истечения срока или выхода.

### 6. **Лимиты и rate limiting**

- **Исправлено:** введён in-memory rate limiting по пользователю (если авторизован) или по IP:
  - `POST /trade/open` — 30 запросов в минуту;
  - `POST /payments/deposit`, `POST /payments/withdraw`, `POST /referral/withdraw` — по 10 запросов в минуту.
  - При превышении возвращается **429** и сообщение «Слишком много запросов. Попробуйте позже.»  
  - Для нескольких инстансов бэкенда нужен общий store (например Redis).

### 7. **Аудит изменений баланса**

- **Исправлено:** таблица `BalanceAuditLog` — каждая операция, меняющая баланс, пишется в журнал:
  - `trade_open` — списание при открытии сделки;
  - `trade_win` — начисление выигрыша;
  - `deposit` — пополнение по колбеку HighHelp;
  - `withdraw` — списание при создании заявки на вывод;
  - `withdraw_refund` — возврат при отказе/ошибке вывода;
  - `referral_transfer` — перевод реферального баланса в основной.
  - Поля: `userId`, `type`, `amount` (дельта), `balanceBefore`, `balanceAfter`, `refType`, `refId`, `createdAt`. Админ может строить отчёты и расследовать инциденты по этим данным.

## PostHog и аудит баланса

**PostHog** — это продуктовая аналитика (события, воронки, записи сессий, feature flags). Он полезен для:
- понимания поведения пользователей (какие экраны открывают, как часто торгуют, отваливаются ли на пополнении);
- обнаружения аномалий (всплески действий, подозрительные паттерны).

**PostHog не заменяет серверный аудит баланса**, потому что:
1. События PostHog отправляются с клиента или через ваш бэкенд «по желанию» — их можно не отправить, подделать или потерять при сбоях.
2. Для финансов и расследований нужен **непререкаемый** источник правды: кто, когда и на какую сумму изменил баланс. Это даёт только серверная запись в БД в момент операции (таблица `BalanceAuditLog`).
3. В аудите баланса важны точные суммы, баланс до/после и привязка к сделке/платежу — это есть в `BalanceAuditLog` и не дублируется в PostHog.

**Итог:** PostHog дополняет аналитику и мониторинг продукта; для аудита баланса и расследований используйте `BalanceAuditLog` (и при необходимости экспортируйте в аналитику или дашборды).

## Telegram: уведомления и аномалии

При заданных в `.env` переменных **TELEGRAM_BOT_TOKEN** и **TELEGRAM_CHAT_ID** бэкенд отправляет в указанный чат (группа или канал):

1. **Каждое изменение баланса** — короткое сообщение: user, тип операции, сумма, баланс до/после, ref (сделка/платёж).
2. **Алерт при аномалии** — отдельное сообщение с префиксом «⚠️ АНОМАЛИЯ», если сработал хотя бы один критерий:
   - **Отрицательный баланс** — в нормальной логике не должен появляться; возможен взлом или прямой доступ к БД.
   - **Крупная операция** — одна операция ≥ порога (по умолчанию 100 000; `TELEGRAM_ANOMALY_LARGE_AMOUNT`).
   - **Много операций за 5 минут** — по одному пользователю (по умолчанию ≥ 15; `TELEGRAM_ANOMALY_RAPID_COUNT`).
   - **Большой приход за 1 минуту** — сумма поступлений за минуту ≥ порога (по умолчанию 50 000; `TELEGRAM_ANOMALY_SPIKE_AMOUNT`).

Пороги можно переопределить в `.env`. Отправка выполняется после записи в БД и не блокирует ответ пользователю. При отсутствии токена/чата уведомления не отправляются.
